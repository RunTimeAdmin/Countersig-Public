AgentID 2.0 — Code Review and Security Analysis
Repository: https://github.com/RunTimeAdmin/AgentID-2.0 Commit reviewed: main HEAD (single commit, "Initial commit") Scope: backend (Node/Express), frontend (React/Vite), database schema, deployment
Context for this review
The repo already contains AgentID_Code_Review.md and AgentID_Security_Audit.md. Both target AgentID 1.0 (April/May 2025) and predate every new subsystem in 2.0: multi-tenant orgs, users + JWT, API keys, RBAC, audit logs with hash chaining, policy engine, webhooks. The cryptographic core (Ed25519 challenge-response) is largely unchanged from 1.0 and remains correctly implemented. This review focuses on the 2.0 additions, which is where almost every new vulnerability lives.


Headline numbers from this review:


* 8 Critical findings
* 9 High findings
* 15 Medium findings
* 11 Low findings


The 1.0 critical bugs (bags x-api-key header, widget URL, env var validation, reporter pubkey unverified) are all fixed in 2.0. Good. New problems took their place.

CRITICAL
C1. Webhook SSRF — internal network is reachable from any admin
Location: backend/src/routes/webhooks.js lines 52, 91; backend/src/services/webhookService.js lines 32-39.


The only validation on webhook.url is url.startsWith('https://'). That blocks http:// and file:// and not much else. A malicious admin (or a stolen admin session) registers a webhook pointing to:


https://169.254.169.254/latest/meta-data/iam/security-credentials/


https://internal-elb.aws.local/admin


https://10.0.0.1:8200/v1/secret/data/anything


https://localhost-rebind.example.com/   (DNS rebinding to 127.0.0.1)


Then the attacker triggers any event the webhook subscribes to. axios.post(webhook.url, body, ...) runs on your server, hits the internal target, and you get back any response that fits in axios's default. Even better for the attacker: Promise.allSettled is fired and forgotten, so failures are invisible to them but the connection still happens.


The HMAC signature is irrelevant here. The attacker controls the URL; they can verify whatever signature your server sends.


Fix:


// In a new utils/url.js


const dns = require('dns').promises;


const net = require('net');


const PRIVATE_RANGES = [


  /^10\./,


  /^172\.(1[6-9]|2\d|3[0-1])\./,


  /^192\.168\./,


  /^127\./,


  /^169\.254\./,


  /^0\./,


  /^::1$/,


  /^fc00:/i,


  /^fe80:/i


];


async function assertPublicHttpsUrl(rawUrl) {


  const u = new URL(rawUrl);                              // throws on garbage


  if (u.protocol !== 'https:') throw new Error('https only');


  if (u.username || u.password) throw new Error('userinfo not allowed');


  const host = u.hostname;


  const addrs = await dns.lookup(host, { all: true });    // resolve once


  for (const { address } of addrs) {


    if (net.isIP(address) === 0) continue;


    if (PRIVATE_RANGES.some(re => re.test(address))) {


      throw new Error(`URL resolves to non-routable address: ${address}`);


    }


  }


  return { url: u.toString(), addrs };                    // pin these IPs


}


Then in webhookService.deliverWebhook, re-validate at delivery time and pin the resolved IP into axios config (use a custom httpsAgent with lookup) so DNS rebinding can't swap the target between validation and delivery. Also set maxRedirects: 0 and maxContentLength/maxBodyLength to small values; right now a target can stream a 10GB response and OOM the worker.


Impact: cloud credential theft, internal API access from the AgentID host, lateral movement. Effort: 4-6 hours including tests and the IP pinning.

C2. JWT_SECRET silently falls back to a hardcoded string
Location: backend/src/services/authService.js line 11.


const JWT_SECRET = process.env.JWT_SECRET || 'dev-jwt-secret-change-in-production';


server.js does check for required env vars, but JWT_SECRET is in the "recommended" list (line 18), not the "required" list. A misconfigured staging/preview environment, a Heroku review app, a CI runner, or any container that lost an env var ends up issuing tokens signed with a string anybody who's read the GitHub repo knows. Anyone can mint themselves an admin token for any org.


Worse: docker-compose.yml line 58 ships a hardcoded JWT_SECRET: local-dev-jwt-secret-change-in-production. If anyone uses the compose file as a starting point and forgets to change that, they're shipping a known secret.


Fix:


// authService.js


const JWT_SECRET = process.env.JWT_SECRET;


if (!JWT_SECRET) {


  throw new Error('JWT_SECRET is required');


}


if (JWT_SECRET.length < 32) {


  throw new Error('JWT_SECRET must be at least 32 characters');


}


And move JWT_SECRET to the required list in server.js. While you're there, validate that it's not the default string in production. Docker compose should use ${JWT_SECRET:?error} syntax to fail at compose time if the host env doesn't supply it.


Impact: total auth bypass. Effort: 15 minutes.

C3. Cross-tenant agent listing and discovery via unauthenticated routes
Location: backend/src/routes/agents.js lines 25, 124.


router.get('/agents', optionalAuth, defaultLimiter, async (req, res, next) => {


  // ...


  const orgId = req.user ? req.user.orgId : null;       // null when unauth


  const agents = await listAgents({ ..., orgId });       // orgId passed as filter


In queries.js listAgents() line 152, the orgId filter is only applied if truthy:


if (orgId) {

So an unauthenticated request to GET /agents?limit=100 returns the top 100 agents across every org in the system, sorted by bags_score descending. Same for GET /discover?capability=anything. This is your tenant boundary, fully bypassed by simply not sending a cookie.

For the public directory case, add an explicit GET /public/agents route that filters WHERE org_id IN (SELECT id FROM organizations WHERE public_directory = true).

C4. Audit log stores raw API keys in plaintext via changes: responseBody
Location: backend/src/middleware/auditMiddleware.js lines 105-108, 142.


The middleware overrides res.json to capture the response body, then writes it into audit_logs.changes for every mutating request. The response body of POST /api-keys is:


return res.status(201).json({


  id: apiKey.id,


  rawKey,                 // <-- the actual unhashed API key


  name: apiKey.name,


  // ...


});


So every API key creation event lands in the audit table with the full plaintext key in the changes column. Indefinitely. Visible to anyone with GET /audit/logs access (managers and above) plus anyone with database read access (DBAs, backups, replicas, anyone who restores a snapshot).


This converts the audit log from a defensive mechanism into a credential trove.


Fix the symptom and the root cause. Symptom: scrub sensitive fields before logging. Root cause: never log full response bodies; log only what you need.


// auditMiddleware.js


const SENSITIVE_FIELDS = new Set(['rawKey', 'password', 'secret', 'refreshToken', 'accessToken']);


function scrub(obj) {


  if (obj === null || typeof obj !== 'object') return obj;


  if (Array.isArray(obj)) return obj.map(scrub);


  const out = {};


  for (const [k, v] of Object.entries(obj)) {


    out[k] = SENSITIVE_FIELDS.has(k) ? '[REDACTED]' : scrub(v);


  }


  return out;


}


// Then:


changes: scrub(responseBody)


Better: don't store the full response. Store a small structured record per action (resource id, fields changed, count). Right now the audit log is also storing every successful login response (which contains user role/orgId — less catastrophic but still reveals account structure to anyone with audit read access).


Same review needed for: webhook secret reveal on POST /orgs/:orgId/webhooks (returns full secret in body), agent registration response.


Impact: full credential exposure to anyone with audit read access; permanent leak in backups. Effort: 1-2 hours including migration to scrub existing rows.

C5. Audit hash chain doesn't actually cover the audited content
Location: backend/src/services/auditService.js lines 84-93.


The "tamper-evident" hash chain is computed over only four fields:


const hashPayload = JSON.stringify({


  action,


  resource_id: resourceId,


  actor_id: actorId,


  timestamp


});


const entryHash = computeHash(prevHash + hashPayload);


Not in the hash: org_id, actor_type, resource_type, changes, metadata, risk_score. An attacker with database write access can do all of the following without breaking the chain:


* Rewrite the changes field of any entry (replace evidence of what was actually done).
* Move entries between orgs by changing org_id.
* Downgrade the risk_score on any sensitive action.
* Modify metadata (IP, user agent, statusCode) to make a hostile login look benign.


verifyAuditChain() will report valid: true because it recomputes the hash using the same four fields. The chain proves "an action of this type happened against this resource at this timestamp" and nothing else.


Fix: hash the entire row payload. Pin the schema:


function buildHashPayload(entry) {


  return JSON.stringify({


    org_id: entry.orgId,


    actor_id: entry.actorId,


    actor_type: entry.actorType,


    action: entry.action,


    resource_type: entry.resourceType,


    resource_id: entry.resourceId,


    changes: entry.changes,            // post-scrubbing


    metadata: entry.metadata,


    risk_score: entry.riskScore,


    timestamp: entry.timestamp


  });


}


Use a deterministic JSON serializer (e.g. safe-stable-stringify) so key ordering doesn't change the hash. Update verifyAuditChain to use the same canonical form. Document the canonical form so the chain remains verifiable across versions.


Optional but worth it: sign each entry hash with a server-side HMAC key separate from JWT_SECRET, so even an attacker with full DB write can't recompute valid hashes without also stealing that key.


Impact: claimed tamper protection is materially weakened; compliance posture ("we have an immutable audit log") is misleading. Effort: 4 hours including data migration and verifier update.

C6. Update message signature doesn't bind to the field changes
Location: backend/src/routes/agents.js lines 225, 263-300.


The signed message is:


const message = `AGENTID-UPDATE:${agentId}:${timestamp}`;


The actual update payload (name, description, capabilities, creatorX, etc.) is read from the request body and applied separately. The signature only proves "the owner authorized SOME update at timestamp T." It does not prove the owner authorized THESE specific changes.


Within the 5-minute window, an attacker who captures one valid signed update on the wire (or recovers it from a leaked log, or pulls it from a misconfigured proxy) can replay it with completely different field values. They keep the signature, swap the body. Server validates signature against the message (which doesn't include the body), accepts it.


Same bug pattern in the /agents/:agentId/revoke route (line 387) — the message format is AGENTID-REVOKE:{agentId}:{timestamp} with no other binding. Less dangerous since revoke has no parameters to swap, but the timestamp window means a single signed revoke can be re-fired up to 5 minutes later (which the freshness window technically covers, but only if the body is also rejected on resubmission, which it isn't).


Fix: bind the payload into the signed message.


// Client side


const fieldsHash = crypto


  .createHash('sha256')


  .update(canonicalJSON({ name, description, capabilities, creatorX, tokenMint }))


  .digest('hex');


const message = `AGENTID-UPDATE:${agentId}:${timestamp}:${fieldsHash}`;


Server reconstructs fieldsHash from the actual request body and verifies the signature against the rebuilt message. Now the signature is non-malleable across field changes.


Impact: authenticated update tampering by anyone who can observe one update message. Effort: 1 hour including frontend signing change.

C7. POST /agents/:agentId/attest is unauthenticated
Location: backend/src/routes/attestations.js line 27.


router.post('/agents/:agentId/attest', authLimiter, async (req, res, next) => {


No authenticate middleware. Anyone on the internet can:


1. POST {success: true} 100,000 times to any agent ID and watch the success counter run up.
2. Trigger refreshAndStoreScore() repeatedly on any agent in any org.
3. Skew any agent's bags_score upward without ever proving they interacted with the agent.


The reputation system is the entire value prop and it's open-mic. The flag endpoint (/flag) at least requires a signature from a claimed reporterPubkey, but an attacker can generate Solana keypairs for free, so the flag system also collapses if any single attacker is willing to spend a few minutes:


# Sybil three keypairs, sign three flag messages, get the agent auto-flagged


for i in 1 2 3; do


  solana-keygen new --outfile /tmp/key$i.json --no-bip39-passphrase


  # sign AGENTID-FLAG:{agentId}:{pubkey$i}:{ts}, POST it


done


The auto-flag threshold of 3 unresolved flags (line 170) makes this a 3-key, ~30-second attack on any agent's reputation across orgs.


Fix:


* Attest: require authentication and check that the calling user belongs to the same org as the agent (or accept signed attestations from the agent's own keypair, similar to the verify flow).
* Flag: add per-org flag rate limits, require flagger to be authenticated (their org user), and decouple the auto-flag from a hard threshold. Use weighted scoring where flags from inside the org count more than external flags. Better: don't auto-flag; queue for review.


Impact: full reputation system manipulation by unauthenticated attackers. Effort: 2-3 hours.

C8. API key auth and RBAC are disconnected — scoped keys cannot pass any authorization check
Location: backend/src/middleware/authenticate.js line 68; backend/src/middleware/authorize.js line 33.


authenticate.js sets req.user.role = 'api_key' for API-key-authenticated requests. authorize.js looks up the role in ROLE_HIERARCHY:


const ROLE_HIERARCHY = {


  admin: 4,


  manager: 3,


  member: 2,


  viewer: 1


};


// ...


const userRoleLevel = ROLE_HIERARCHY[req.user.role] || 0;


'api_key' is not in the hierarchy, so it gets level 0. Every route that uses authorize(ROLES.X) rejects API key requests, regardless of the scopes you assigned to the key. The scope JSON column on api_keys is set, stored, listed in the UI, and never enforced.


Effectively: API keys can only call routes that don't use authorize(). Right now those routes are mostly the public-ish read-only ones plus the un-authenticated attest route. Programmatic admin via API key is impossible.


Either model needs to be fixed. The cleaner model is: API keys inherit the role of the user who created them, optionally narrowed by scopes. Store both role and scopes on the key row. Pass them through to req.user:


// authenticate.js, when API key matches


req.user = {


  userId: apiKey.user_id,


  orgId: apiKey.org_id,


  role: apiKey.role,           // store role on the key


  scopes: apiKey.scopes,


  apiKeyId: apiKey.id,


  isApiKey: true               // for any code that needs to differentiate


};


Then authorize() works the same way for both auth modes, and a new requireScope('agents:write') middleware can enforce scope-level restrictions on top.


Right now your API key product is non-functional for write operations. This is the kind of bug a customer hits 30 seconds after signing up.


Impact: API keys are effectively read-only despite advertising scopes. Effort: 3-4 hours including migration to add role to api_keys table.

HIGH
H1. No CSRF protection on cookie-authenticated routes
Location: backend/server.js (no CSRF middleware); backend/src/routes/auth.js line 24 (sameSite: 'lax').


Auth uses HttpOnly cookies. sameSite: 'lax' blocks most cross-site form posts but allows top-level navigation requests (clicking a link), which means:


* An attacker hosts evil.com with a form that POSTs to https://agentid2.provenanceai.network/api-keys.
* A logged-in admin clicks "click here for free Bitcoin."
* Browser sends the cookie because it's a top-level navigation.
* New API key is created in the admin's org.


The browser-only attack surface is limited (lax blocks subresource POSTs and AJAX), but the navigation pivot is real for any state change that can be triggered with a top-level form submission.


Fix: double-submit CSRF token, or SameSite: 'strict', or use CORS preflight as a CSRF gate (require a custom header like X-Requested-With and verify it server-side; lax-mode browsers don't include custom headers on top-level navigation).


// Cheap fix: require X-Requested-With on all mutating routes


app.use((req, res, next) => {


  if (['POST', 'PUT', 'DELETE', 'PATCH'].includes(req.method)) {


    if (req.get('X-Requested-With') !== 'AgentID') {


      return res.status(403).json({ error: 'CSRF check failed' });


    }


  }


  next();


});


And configure the frontend axios client to set that header.


Impact: CSRF on state-changing routes; depends on attacker's ability to lure an admin to a hostile page. Effort: 1-2 hours.

Location: backend/src/routes/auth.js lines 154-163.


if (result.rows.length === 0) {


  return res.status(401).json({ error: 'Invalid email or password' });   // fast path

H3. No login attempt tracking, no account lockout
Location: backend/src/routes/auth.js (no failed attempt counter); users table schema (no failed_login_count column).


/auth/login has no per-account rate limit, no lockout, no exponential backoff. The only barrier is the global defaultLimiter (100 req/15min per IP). For credential stuffing with a botnet that distributes across thousands of IPs, this is no defense.


Failed logins also aren't audited because the audit middleware skips entries without req.user.orgId, which is unset on login failure (login failure happens before the cookie is issued). So there's no detective control either.


Fix: add failed_login_count and locked_until to users. On bad password, increment the counter. After 5 failures, lock the account for 15 minutes and require email-based unlock. Audit failed logins explicitly inside the route handler (don't rely on the middleware).


Impact: credential stuffing attacks succeed against weak passwords; no detection. Effort: 2-3 hours including migration.

H5. Refresh token comparison is not constant-time
Location: backend/src/services/authService.js line 153.


const stored = await redis.get(`refresh:${userId}`);


return stored === refreshToken;


JavaScript === on strings short-circuits at the first differing character. The timing difference is small but observable across thousands of network requests, especially in a co-located attacker scenario. JWTs are signed (so even forging one is hard), but the constant-time comparison is free defense in depth.


Fix:


const crypto = require('crypto');


function safeEqual(a, b) {


  if (typeof a !== 'string' || typeof b !== 'string') return false;


  const ab = Buffer.from(a, 'utf8');


  const bb = Buffer.from(b, 'utf8');


  if (ab.length !== bb.length) return false;


  return crypto.timingSafeEqual(ab, bb);


}


return safeEqual(stored, refreshToken);


Impact: small. Worth fixing because it's a reflex pattern, not a difficult one. Effort: 5 minutes.

H6. Single-key refresh token model breaks concurrent sessions and creates rotation race
Location: backend/src/services/authService.js lines 121, 136.


Refresh tokens are stored with the key refresh:${userId} — one slot per user. Consequences:


* Logging in on a second device kicks the first device off (the second setex overwrites).
* Rotation race: /auth/refresh does revokeRefreshToken(userId) then storeRefreshToken(userId, newToken). If two refresh requests arrive in close succession (frontend tab + service worker), the second storeRefreshToken may complete first, then the first request's revoke deletes the new token, logging the user out.
* No way to revoke a single device's session without revoking all of them.


Fix: index by token id, not user id. Generate a tokenId per refresh token (UUID, embedded in the JWT as jti), store refresh:${tokenId} with value userId, and verify by token id on refresh:


function generateTokens(user) {


  const tokenId = crypto.randomUUID();


  const refreshToken = jwt.sign(


    { userId: user.id, type: 'refresh', jti: tokenId },


    JWT_SECRET,


    { expiresIn: JWT_REFRESH_EXPIRY }


  );


  return { accessToken: ..., refreshToken, tokenId };


}


async function storeRefreshToken(tokenId, userId) {


  await redis.setex(`refresh:${tokenId}`, REFRESH_TTL_SECONDS, userId);


}


This also enables a per-user "logout all sessions" by maintaining a user_sessions:${userId} set of token ids and deleting them all.


Impact: bad UX (concurrent session breakage), occasional spurious logouts on rotation race. Effort: 3-4 hours.

H7. Frontend reads tokens from localStorage — XSS can steal them
Location: frontend/src/lib/api.js line 14.


const token = localStorage.getItem('agentid_token');


if (token) {


  config.headers.Authorization = `Bearer ${token}`;


}


The backend uses HttpOnly cookies, which is the secure pattern. The frontend interceptor reads agentid_token from localStorage and adds it as a Bearer header anyway. If anything ever writes a token there (legacy code, partial implementation, future regression), it becomes XSS-stealable.


AuthProvider.jsx also stores user state in localStorage.getItem('agentid_user') — that's just user metadata and probably fine, but the agentid_token slot is a foot-gun.


Fix: remove the localStorage read. The HttpOnly cookies handle session auth. For API key use cases (like a CLI tool), require explicit configuration rather than reading from localStorage where XSS can find it.


api.interceptors.request.use((config) => {


  config.withCredentials = true;        // send the HttpOnly cookies


  return config;


});


Impact: latent XSS-to-account-takeover vector. Effort: 15 minutes.

H8. SERIALIZABLE + FOR UPDATE on every audit log insert creates a write bottleneck
Location: backend/src/services/auditService.js lines 71, 75.


await client.query('BEGIN ISOLATION LEVEL SERIALIZABLE');


const prevResult = await client.query(


  'SELECT entry_hash FROM audit_logs WHERE org_id = $1 ORDER BY id DESC LIMIT 1 FOR UPDATE',


  [orgId]


);


Every audit log insert serializes against every other audit log insert in the same org. Under modest load (50 RPS for a busy customer), you'll see serialization failures, retry storms, and end-of-quarter audit burst pain.


The FOR UPDATE does correctly prevent two concurrent inserts from both reading the same prevHash and creating a fork in the chain, so there's a real correctness reason for the lock. But SERIALIZABLE on top is overkill for this read-and-write pattern.


Fix:


* Drop the SERIALIZABLE isolation. READ COMMITTED (Postgres default) is sufficient when you have the explicit FOR UPDATE row lock.
* Consider moving audit log writes to an async queue (Redis stream, Kafka, or even an in-process queue with bounded concurrency) so the request path isn't blocked on the lock.
* For bulk operations (e.g. policy engine triggering 1,000 revokes), batch into a single audit entry rather than one per agent.


await client.query('BEGIN');                              // default READ COMMITTED


const prevResult = await client.query(


  'SELECT entry_hash FROM audit_logs WHERE org_id = $1 ORDER BY id DESC LIMIT 1 FOR UPDATE',


  [orgId]


);


Impact: scaling ceiling on audit-heavy orgs. Effort: 30 minutes for the isolation change; 1-2 days for the async queue.

H9. JWTs lack aud and iss claims
Location: backend/src/services/authService.js lines 56-66.


The signed JWT contains userId, email, orgId, role, and type. No aud (audience) or iss (issuer). If JWT_SECRET is ever reused across services (the same Anthropic-style "platform key"), tokens issued by service A pass verification on service B. verifyAccessToken doesn't even check iss/aud so even if you set them, they'd be ignored.


Fix:


function generateTokens(user) {


  const opts = {


    expiresIn: JWT_EXPIRY,


    issuer: 'agentid.io',


    audience: 'agentid-api'


  };


  const accessToken = jwt.sign({ ... }, JWT_SECRET, opts);


  // ...


}


function verifyAccessToken(token) {


  const decoded = jwt.verify(token, JWT_SECRET, {


    issuer: 'agentid.io',


    audience: 'agentid-api'


  });


  // ...


}


Impact: low until you scale to multiple services with shared secrets; cheap defense in depth. Effort: 15 minutes.

M3. Any authenticated user can create or revoke API keys

M5. Missing partial indexes on filter columns
Location: backend/src/models/migrate-v2.js.


The listAgents query filters by org_id AND status AND is_demo = false AND revoked_at IS NULL. There's an idx_agent_identities_org on (org_id), but no composite or partial index. For a 1M-row agent table, the planner will use the org_id index then heap-scan the rest. Add:

  ON agent_identities(org_id, status, bags_score DESC)

CREATE INDEX IF NOT EXISTS idx_policy_rules_active

CREATE INDEX IF NOT EXISTS idx_audit_logs_org_created

M6. No CHECK constraints on enum-like columns
Location: backend/src/models/migrate-v2.js.


users.role, agent_identities.status, webhooks.enabled, policy_rules.action are all VARCHAR with application-side validation only. A bug or a direct DB write can put 'super_admin' in users.role or 'execute_arbitrary_code' in policy_rules.action and the application will silently treat it as no-op (level 0 in the hierarchy, "unknown action" in the engine).


Fix:


ALTER TABLE users ADD CONSTRAINT users_role_check

ALTER TABLE agent_identities ADD CONSTRAINT agents_status_check


  CHECK (status IN ('active', 'flagged', 'revoked', 'disabled'));


ALTER TABLE policy_rules ADD CONSTRAINT policy_action_check


  CHECK (action IN ('revoke', 'flag', 'notify', 'disable'));

M7. No CASCADE behavior on org delete
Location: backend/src/models/migrate-v2.js.


organizations has child tables (users, api_keys, audit_logs, policy_rules, webhooks, agent_identities) with FK references but no ON DELETE clause. Deleting an org will fail with FK constraint errors. Decide your model:


* Hard delete with cascade: ON DELETE CASCADE on every FK.
* Soft delete with propagation: filter WHERE deleted_at IS NULL on org joins, and have a scheduled cleanup that hard-deletes tombstoned orgs after a retention period.

M11. Policy engine evaluates rules sequentially in event listener
Location: backend/src/services/policyEngine.js lines 156-164.


for (const rule of result.rows) {


  const matches = evaluateCondition(rule.condition, event);


  if (matches) {


    const actionResult = await executeAction(rule, event);  // sequential


    triggered.push(...);

}


If an org has 200 enabled rules and an event matches 50 of them, every action waits on the previous (each action is a DB write or an event publish). Under load, the event listener accumulates back-pressure.


Fix: parallelize matching actions with Promise.allSettled:


const matched = result.rows.filter(rule => evaluateCondition(rule.condition, event));

);


Better: move policy execution to a queue worker, not the event listener.

M15. No max payload size or response size limit on webhook deliveries
Location: backend/src/services/webhookService.js lines 32-39.


Axios defaults: no maxContentLength, no maxBodyLength, follows up to 5 redirects, no response size cap. A malicious webhook target can:


* Stream a 10 GB response and OOM your worker.

const response = await axios.post(webhook.url, body, {

  timeout: 10000,

});

What's gone right
It's worth naming the things this codebase does correctly, since this review is a long list of what to change:


* All SQL is parameterized. No string concatenation, no template literals into queries. The dynamic update builder in queries.js updateAgent correctly uses positional parameters.
* Ed25519 challenge-response is implemented correctly with tweetnacl. Single-use nonces, expiry, base58 encoding all consistent.
* HTML escaping in badgeBuilder.js (both escapeHtml and escapeXml) is applied to user-controlled fields in the widget output.
* Tenant isolation via orgContext middleware is enforced on the routes that use it (orgs, policies, webhooks, audit).
* Refresh token rotation on /auth/refresh is the right pattern, even if H6 needs to be addressed.
* Hash-chained audit log is the right idea, even if C5 needs the chain to actually cover the data.
* The bagsAuthVerifier x-api-key bug from the 1.0 review is fixed.
* The widgetUrl base URL bug from the 1.0 review is fixed (uses config.agentIdBaseUrl).
* Required env var validation at startup exists and exits cleanly with a helpful message.
* Test scaffolding covers the cryptographic services and the policy engine (~1650 lines).


The architecture is sound. Most of the criticals are application-layer bugs from a complex multi-tenant addition, not foundational design problems.

Suggested feature additions (separate from fixes)
Beyond bug fixes, things that would materially strengthen the product:


1. Agent identity rotation: cryptographic key rotation flow that allows an agent owner to migrate from pubkey_old to pubkey_new without losing their reputation score. Sign a binding message with both keys, server validates both signatures, updates the row. Closes the "lost key = lost identity" gap.


2. Per-environment config visibility: a /health/config endpoint (admin-only) that reports which env vars are set without revealing values. Helps debug missing config in staging.


3. Webhook delivery dashboard: per-webhook view of last 100 deliveries with status, retry status, response codes. Combined with M12 above, this is the difference between "AgentID is broken" and "your endpoint returned 500 yesterday at 3pm."


4. Org-level usage metrics: rate of registrations, verifications, attestations per org per day. Useful for both the operator (capacity planning) and the customer (knowing they're under attack).


5. Bring-your-own-key for SAID binding: today the binding posts to a single SAID gateway. Allow orgs to register their own gateway URL for private/permissioned identity registries.


6. Anomaly detection on attestation flow: if an agent's success-rate drops 50% in 1 hour, flag for review. Currently the attest endpoint accepts whatever and the score updates without context.


7. Public widget customization: today every widget looks the same. Let orgs upload a small theme JSON (colors, logo URL) so embedded widgets match the host site's design.


8. SBOM and dependency scanning in CI: this is a security product. Running npm audit and osv-scanner on every PR is table stakes. The current package.json has reasonably current deps but no automated check.


9. Secret scanning on commits: gitleaks or GitHub's native secret scanning. The hardcoded secrets in docker-compose.yml would have flagged.


10. Postgres row-level security (RLS): you're already filtering by org_id in queries. RLS makes that filter mandatory at the database level, so a future bug (someone forgets the filter) doesn't leak data. It's a real engineering investment but it's the best defense for a multi-tenant SaaS.
