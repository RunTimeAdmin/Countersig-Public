Executive Summary</h2><p>AgentID is a trust verification registry for AI agents operating on the Bags/Solana ecosystem. Its core security model is based on <strong>Ed25519 asymmetric cryptography</strong> — the same battle-tested primitive used by SSH, Solana, and Signal.</p><p>The fundamental security guarantee AgentID provides is this:</p><blockquote> <p><strong>Private keys are never transmitted to, requested by, or stored in AgentID at any point in the registration or verification lifecycle.</strong></p> </blockquote><p>This audit confirms that guarantee holds in the current codebase. The cryptographic core is sound. The identified vulnerabilities are in the <strong>trust metadata layer</strong> — the information built on top of the cryptographic proof — not in the cryptographic proof itself.</p><p><strong>Overall Security Rating: B+ (Strong Core, Known Metadata Trust Gaps)</strong></p><p>The project is appropriate for hackathon deployment and controlled production use. Three items should be addressed before handling high-value agent reputations at scale.</p><hr><h2>2. </code></pre><p>Nonces are one-time-use random strings issued during the challenge-response flow. They expire automatically. A nonce without the corresponding private key is cryptographically useless — it cannot be used to forge a signature.</p><h4>Badge Cache</h4><pre><code>Key:    badge:{pubkey}

TTL:    Configurable (default ~60 seconds)

No sensitive data.</p><p><strong>Worst case if Redis is fully exposed:</strong> An attacker sees active challenge nonces and cached badge SVGs. The nonces are useless without private keys. Cryptographic Model</h2><h3>3.1 Ed25519 Overview</h3><p>AgentID uses <strong>Ed25519</strong> — an elliptic curve signature scheme offering:</p><ul> <li>128-bit security level (equivalent to RSA-3072)</li> <li>Fast signature generation and verification</li> <li>Small key sizes (32-byte public key, 64-byte secret key)</li> <li>Deterministic signatures (no random number generator dependency during signing)</li> <li>Resistance to timing attacks</li> </ul><p>This is the same algorithm used by:</p><ul> <li>SSH (OpenSSH default since 2014)</li> <li>Solana (all wallet keypairs)</li> <li>Signal Protocol</li> <li>TLS 1.3</li> </ul><h3>3.2 The Challenge-Response Flow</h3><p>The registration and verification flow is a standard <strong>Sigma protocol</strong> — a zero-knowledge proof of private key ownership.</p><pre><code>Step 1 — Challenge Issuance
  AgentID generates:  nonce = cryptographically random 32-byte string
  AgentID stores:     {nonce, pubkey, expiresAt} in Redis (TTL: 5 min)
  AgentID returns:    message = "AGENTID-VERIFY:{pubkey}:{nonce}:{timestamp}"

Step 2 — Agent Signs
  Agent computes:     signature = Ed25519.sign(message, privateKey)
  Agent sends:        {pubkey, nonce, signature}  ← private key NOT sent

Step 3 — AgentID Verifies
  AgentID computes:   valid = Ed25519.verify(signature, message, pubkey)
  AgentID checks:     nonce exists in Redis and has not expired
  AgentID deletes:    nonce from Redis (prevents replay)
  If valid:           registration proceeds
  If invalid:         401 Unauthorized
</code></pre><h3>3.3 Why This Cannot Be Faked</h3><p><strong>Scenario: Attacker knows InfraWatch's pubkey and tries to register as InfraWatch</strong></p><pre><code>Attacker sends:     pubkey = DVKMPX3JFyfR7LPZECU7DVqYNeHc26t2FRRpC3rgWCYP
AgentID issues:     nonce = "a7f3k9..." (random, never seen before)
                    message = "AGENTID-VERIFY:DVKMPX3...:a7f3k9...:1714500000"
Attacker must:      produce valid Ed25519 signature for this exact message
Without:            InfraWatch's 64-byte private key
Result:             Mathematically impossible with current computing technology
                    (Would require ~2^128 operations to brute force)
</code></pre><p><strong>Scenario: Attacker replays a previously observed signature</strong></p><pre><code>Attacker captures:  old signature from a previous valid verification
Attacker replays:   same {pubkey, nonce, signature}
AgentID checks:     nonce "a7f3k9..." in Redis → NOT FOUND (already consumed)
Result:             404 / 401 — replay rejected
</code></pre><p><strong>Scenario: Attacker intercepts the challenge message in transit</strong></p><pre><code>Attacker intercepts: message = "AGENTID-VERIFY:DVKMPX3...:a7f3k9...:1714500000"
Attacker must:       sign it with InfraWatch's private key
Without:             the private key → cannot produce valid signature
Result:              Useless information
</code></pre><h3>3.4 Implementation: tweetnacl</h3><p>AgentID uses the <code>tweetnacl</code> library for all cryptographic operations:</p><pre><code class="language-javascript">// Verification (server-side, verify.js / bagsAuthVerifier.js)
nacl.sign.detached.verify(messageBytes, signatureBytes, publicKeyBytes)
</code></pre><p><code>tweetnacl</code> is:</p><ul> <li>Audited and widely used (downloaded ~3M times/week on npm)</li> <li>A direct JavaScript port of the original TweetNaCl C implementation by Bernstein et al.</li> <li>Constant-time — resistant to timing side-channel attacks</li> <li>No dependencies</li> </ul><hr><h2>4. Attack Surface Analysis</h2><h3>4.1 ✅ Private Key Exposure — NOT POSSIBLE via AgentID</h3><p><strong>Threat:</strong> An attacker compromises AgentID's database or API and obtains agent private keys.</p><p><strong>Assessment:</strong> Not possible by design. Private keys are never requested, transmitted, or stored anywhere in the AgentID system. The challenge-response protocol was specifically designed to prove key ownership without key disclosure.</p><p><strong>Evidence in codebase:</strong></p><ul> <li><code>register.js</code> — accepts <code>signature</code> and <code>message</code>, never <code>privateKey</code></li> <li><code>verify.js</code> — accepts <code>pubkey</code>, <code>nonce</code>, <code>signature</code>, never <code>privateKey</code></li> <li><code>queries.js</code> — no <code>private_key</code> column in any table schema</li> <li><code>bagsAuthVerifier.js</code> — verifies signatures, never stores them</li> </ul><p><strong>Risk Level: 🟢 None</strong></p><hr><h3>4.2 ✅ Challenge Replay Attack — MITIGATED</h3><p><strong>Threat:</strong> An attacker captures a valid challenge-response pair and replays it to authenticate as the agent later.</p><p><strong>Mitigations in place:</strong></p><ol> <li><strong>Nonce uniqueness</strong> — every challenge has a cryptographically random nonce</li> <li><strong>Nonce expiry</strong> — Redis TTL of 5 minutes; expired nonces rejected</li> <li><strong>Nonce consumption</strong> — nonce deleted from Redis immediately after use; cannot be used twice</li> <li><strong>Timestamp binding</strong> — message includes timestamp; stale messages detectable</li> </ol><p><strong>Risk Level: 🟢 Mitigated</strong></p><hr><h3>4.3 🔴 False Flag Attack — KNOWN VULNERABILITY (OPEN)</h3><p><strong>Threat:</strong> An attacker submits malicious flags against legitimate agents to damage their reputation scores without owning the reporter pubkey they claim.</p><p><strong>Current behavior:</strong></p><pre><code>POST /attestations/flag
{
  "agentPubkey": "DVKMPX3...",   ← target agent
  "reporterPubkey": "ANYKEY...", ← self-asserted, not verified
  "reason": "scam agent"
}
</code></pre><p>No signature is required from the reporter. Anyone can file a flag claiming any pubkey as the reporter.</p><p><strong>Impact:</strong> Reputation score manipulation. An attacker could systematically flag legitimate agents, lowering their <code>bags_score</code> and reducing their visibility in discovery results.</p><p><strong>Current rate limiting:</strong> <code>defaultLimiter</code> is applied — less restrictive than <code>authLimiter</code> used on the verify endpoints. This is a secondary issue noted in Sprint 1 (item S2).</p><p><strong>Status:</strong> TODO comment exists in <code>attestations.js</code>. Fix is designed but not yet implemented. See Section 6 for the complete fix design.</p><p><strong>Risk Level: 🔴 Open — Highest priority for Sprint 2</strong></p><hr><h3>4.4 🟡 Self-Asserted Metadata — TRUST LIMITATION</h3><p><strong>Threat:</strong> An agent registers with false metadata — claiming to be a well-known brand, using someone else's Twitter handle, or inflating its capabilities.</p><p><strong>Current behavior:</strong> <code>name</code>, <code>description</code>, <code>creatorX</code>, <code>creatorWallet</code>, and <code>capabilities</code> are all taken at face value. AgentID verifies that you own the keypair — it does not verify that you are who you claim to be in your profile.</p><p><strong>Example attack:</strong></p><pre><code class="language-json">{
  "pubkey": "ATTACKER_PUBKEY",
  "name": "Official OpenAI GPT-5 Agent",
  "creatorX": "openai",
  "description": "The official OpenAI agent on Bags"
}
</code></pre><p>This registration would succeed. The attacker cryptographically proves they own <code>ATTACKER_PUBKEY</code> — but nothing verifies the name or creatorX.</p><p><strong>Mitigations possible but not yet implemented:</strong></p><ul> <li>Twitter OAuth verification of <code>creatorX</code></li> <li>On-chain verification of <code>creatorWallet</code> (sign a message with the wallet)</li> <li>Manual review / community flagging (the flag system, once hardened)</li> <li>SAID registry cross-reference for known agents</li> </ul><p><strong>Risk Level: 🟡 Medium — Acceptable for hackathon; needs social verification layer for production</strong></p><hr><h3>4.5 🟡 No Agent Identity Recovery — KEY MANAGEMENT RISK</h3><p><strong>Threat:</strong> An agent operator loses their private key, or an attacker steals it.</p><p><strong>If private key is lost:</strong></p><ul> <li>The agent identity is permanently inaccessible</li> <li>No "forgot your password" flow exists</li> <li>The registered pubkey becomes a dead entry in the registry</li> <li>Operator must register a new keypair as a new agent identity</li> </ul><p><strong>If private key is stolen:</strong></p><ul> <li>Attacker can pass future challenges as that agent</li> <li>Can re-verify the agent, keeping its score active</li> <li>Cannot change the metadata (requires a new signed request — same key requirement)</li> <li>There is no revocation mechanism currently</li> </ul><p><strong>Risk Level: 🟡 Medium — Standard risk for all Ed25519-based systems; revocation mechanism is a known gap</strong></p><hr><h3>4.6 🟡 Database Breach — STANDARD INFRASTRUCTURE RISK</h3><p><strong>Threat:</strong> Postgres database is compromised by an attacker.</p><p><strong>What is exposed:</strong></p><ul> <li>All agent public keys, names, descriptions, scores — largely public data</li> <li>Creator Twitter handles and wallet addresses — semi-public</li> <li>Flag history — could reveal community disputes</li> </ul><p><strong>What is NOT exposed:</strong></p><ul> <li>Private keys (never stored)</li> <li>Bags API credentials (in .env only)</li> <li>Redis credentials (in .env only)</li> </ul><p><strong>What an attacker could do:</strong></p><ul> <li>Read the full agent registry</li> <li>Directly manipulate reputation scores (bags_score) without going through the API</li> <li>Delete agents from the registry</li> <li>Resolve or create flags directly</li> </ul><p><strong>Mitigations:</strong></p><ul> <li>Postgres should not be publicly accessible — bind to localhost or private network only</li> <li>Use a dedicated low-privilege DB user for the application (not superuser)</li> <li>Regular backups</li> <li>Consider encrypting sensitive columns (creator_x, creator_wallet) at rest</li> </ul><p><strong>Risk Level: 🟡 Medium — Standard database security practices apply</strong></p><hr><h3>4.7 🟢 SAID Gateway Dependency — GRACEFUL DEGRADATION</h3><p><strong>Threat:</strong> SAID Gateway is unavailable or returns incorrect trust scores.</p><p><strong>Current behavior:</strong> SAID binding during registration is non-blocking — if SAID is down, registration proceeds and <code>said_trust_score</code> defaults to 0. Reputation scoring degrades gracefully.</p><p><strong>Risk Level: 🟢 Low — Handled gracefully</strong></p><hr><h3>4.8 🟢 Rate Limiting — MOSTLY ADEQUATE</h3><p><strong>Current state:</strong></p><ul> <li><code>/verify/challenge</code> → <code>authLimiter</code> ✅ (stricter)</li> <li><code>/verify/response</code> → <code>authLimiter</code> ✅ (stricter)</li> <li><code>/register</code> → <code>authLimiter</code> ✅ (stricter)</li> <li><code>/attestations/flag</code> → <code>defaultLimiter</code> ⚠️ (should be authLimiter)</li> <li>All other routes → <code>defaultLimiter</code> ✅ (appropriate)</li> </ul><p><strong>One gap:</strong> The flag submission endpoint uses the less restrictive <code>defaultLimiter</code>. Combined with the unverified reporter pubkey issue (4.3), this makes flag spam easier than it should be.</p><p><strong>Risk Level: 🟢 Low overall — one targeted fix needed on flag endpoint</strong></p><hr><h2>5. How the False Flag Vulnerability Works and How to Fix It</h2><h3>The Problem in Plain English</h3><p>Right now the flag endpoint is like a <strong>complaint box with no ID check.</strong> Anyone can walk up and drop in a complaint claiming to be anyone. The reputation scoring system reads those complaints and lowers the flagged agent's score.</p><h3>The Current Code Flow</h3><pre><code>Client POSTs:   { agentPubkey, reporterPubkey, reason }
Server does:    INSERT INTO agent_flags (agent_pubkey, reporter_pubkey, reason)
Server does:    recalculate bags_score (community factor reduced)
No check:       Was reporter_pubkey actually signed by the reporter?
</code></pre><h3>The Fix Design (Same Pattern Already in the Codebase)</h3><p>The PKI challenge-response flow in <code>verify.js</code> already solves this exact problem for agent registration. The same pattern needs to be applied to flag submission.</p><p><strong>What changes on the client side:</strong></p><pre><code>Before submitting a flag, the reporter must:
1. Sign a payload containing: { agentPubkey, reason, reporterPubkey, timestamp }
2. Include that signature in the flag submission request
</code></pre><p><strong>What changes on the server side:</strong></p><pre><code>Before inserting the flag, the server must:
1. Reconstruct the expected payload
2. Call verifyBagsSignature(payload, signature, reporterPubkey)
3. Only insert the flag if signature is valid
4. Reject with 401 if signature is invalid
</code></pre><p><strong>The payload to sign:</strong></p><pre><code class="language-javascript">const flagPayload = JSON.stringify({
  agentPubkey: "DVKMPX3...",
  reason: "suspicious activity",
  reporterPubkey: "REPORTER_PUBKEY",
  timestamp: Date.now()
});
// Sign flagPayload with reporter's private key
// Send { ...flagData, signature, timestamp } to the API
</code></pre><p><strong>The server verification:</strong></p><pre><code class="language-javascript">// In attestations.js, before INSERT:
const payload = JSON.stringify({
  agentPubkey: req.body.agentPubkey,
  reason: req.body.reason,
  reporterPubkey: req.body.reporterPubkey,
  timestamp: req.body.timestamp
});
const isValid = verifyBagsSignature(payload, req.body.signature, req.body.reporterPubkey);
if (!isValid) return res.status(401).json({ error: 'Invalid reporter signature' });
</code></pre><h3>Why This Works</h3><p>After this fix, to file a valid flag you must:</p><ol> <li>Know the reporter pubkey <strong>AND</strong></li> <li>Have the matching private key <strong>AND</strong></li> <li>Sign the specific payload for this specific flag at this specific timestamp</li> </ol><p>Without all three, the flag is rejected. Flag spam becomes cryptographically impossible without a valid keypair — and flagging with your own real keypair creates accountability (your pubkey is permanently attached to the flag).</p><h3>Additional Enhancement: Timestamp Window</h3><p>Add a timestamp validity check (same as the existing <code>TIMESTAMP_WINDOW_MS</code> pattern):</p><pre><code class="language-javascript">const FIVE_MINUTES = 5 * 60 * 1000;
if (Date.now() - req.body.timestamp &gt; FIVE_MINUTES) {
  return res.status(400).json({ error: 'Flag request expired' });
