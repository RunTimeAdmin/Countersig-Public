The diff is clean and the patterns are right. Partially closed C1 Webhook SSRF — DNS rebinding still possible Location: backend/src/utils/urlValidator.js , backend/src/services/webhookService.js . The assertPublicHttpsUrl validator does the right thing. It's called at registration time and again at delivery time, which closes the basic TOCTOU. It blocks RFC1918, link-local, loopback, and IPv6 ULAs. Solid. What it doesn't fix: DNS rebinding. The flow is: Validator calls dns.lookup(host) , gets 8.8.8.8 , allows. Validator returns. The resolved address is included in the return value but never used. axios.post(webhook.url, ...) is called. Axios does its own DNS lookup. Attacker's DNS server returns a different answer the second time. Could be 127.0.0.1 , could be an internal IP. Axios connects there. The window between step 1 and step 3 is small but nonzero. For an attacker-controlled domain, returning different answers based on TTL or query count is trivial. To actually close this, you need to make the axios call to the resolved IP directly and pass the original hostname as a Host: header (and servername in the TLS config so SNI matches). Or use a custom lookup function in https.Agent that returns the IP from the validator. const https = require('https'); async function deliverWebhook(webhook, event) { const validation = await assertPublicHttpsUrl(webhook.url); const pinnedAddress = validation.resolvedAddresses[0]; const agent = new https.Agent({ lookup: (hostname, options, cb) =&gt; { // Always return the IP we already validated, regardless of DNS cb(null, pinnedAddress, net.isIP(pinnedAddress)); } }); await axios.post(webhook.url, body, { httpsAgent: agent, headers: { ... }, // ... }); } Risk level: medium. The base case (someone registers https://10.0.0.1/admin ) is now blocked. The sophisticated case (rebinding) still works. For an enterprise-targeted product, rebinding is exactly the kind of thing red teams test. Effort: 2 hours including tests against a malicious DNS server. C3 Cross-tenant agent data — /agents/owner/:pubkey still leaks Location: backend/src/routes/agents.js line 116. GET /agents and GET /discover now require authentication. Good. But GET /agents/owner/:pubkey is still using optionalAuth , which means an unauthenticated request returns agents for that pubkey across all orgs (because getAgentsByOwner(pubkey, null) skips the org filter when orgId is null). This is a smaller leak than the original GET /agents issue (it requires knowing a specific pubkey rather than enumerating freely), but it still answers the question &quot;which orgs is this pubkey in&quot; by returning the full set across orgs. Two ways to fix: Require authentication and filter by the caller's org. Same pattern as /agents . Move pubkey-keyed lookup to /public/agents/owner/:pubkey with explicit limited fields, like the existing /public/agents route. Effort: 15 minutes. C6 Update message field binding — legacy fallback negates the fix Location: backend/src/routes/agents.js lines 282-300. The new format with fieldsHash is correct. But the route still falls back to the legacy format ( AGENTID-UPDATE:{agentId}:{timestamp} without the hash) and only logs a console.warn : isSignatureValid = nacl.sign.detached.verify(messageBytes, sigBytes, pubkeyBytes); if (!isSignatureValid) { const legacyBytes = Buffer.from(legacyMessage, 'utf-8'); isSignatureValid = nacl.sign.detached.verify(legacyBytes, sigBytes, pubkeyBytes); if (isSignatureValid) { console.warn('Deprecated signature format used for agent update (no field binding). Agent ID:', agentId); } } An attacker who captured any pre-fix update message can still replay-with-mutation within 5 minutes. They submit the legacy-format signature, the route accepts it, and the body fields get applied without the cryptographic binding the fix was meant to add. The warning log is invisible operationally. Backward compatibility for active agents is reasonable, but it needs an off-switch. Two ways: Option A: hard cutoff date. const LEGACY_DEADLINE = new Date('2026-06-01T00:00:00Z'); if (Date.now() &gt; LEGACY_DEADLINE.getTime()) { // Legacy format rejected entirely } Option B: per-agent flag. Add legacy_signing_disabled BOOLEAN DEFAULT false to agent_identities . The first time an agent uses the new format, set the flag. From then on, the legacy path is rejected for that agent. New agents have the flag set on registration. Option B is cleaner because no agent is forced to migrate at a specific time, but each agent only has the legacy window once. The warning should also be a structured audit log entry, not just console.warn , so you can see which agents are still on the old format. Right now the warning is unwatched and the bypass is open indefinitely. Effort: 2 hours for option B with a small migration. M3 RBAC on API keys — list endpoint missing the check Location: backend/src/routes/apikeys.js line 87. POST /api-keys and DELETE /api-keys/:id now require admin role. GET /api-keys doesn't. Any authenticated user (member, viewer) can list every API key in the org with id , name , keyPrefix , scopes , lastUsed , expiresAt . That's not a credential leak (the keyPrefix is only the first 12 characters), but it does enumerate which admins have which capabilities and when they're being used. Fix: router.get('/api-keys', authorize(ROLES.ADMIN), async (req, res, next) =&gt; { ... }); Or limit non-admins to seeing only keys they personally created. Effort: 5 minutes. M6 CHECK constraints — missing on agent_identities.status Location: backend/src/models/migrate-v2.js line 175. users.role and policy_rules.action got CHECK constraints. agent_identities.status did not. Status values are referenced throughout the codebase as 'active' , 'flagged' , 'revoked' , 'disabled' but nothing enforces it. A bug or direct DB write could set status to anything. Fix: ALTER TABLE agent_identities ADD CONSTRAINT agents_status_check CHECK (status IN ('active', 'flagged', 'revoked', 'disabled')); Make sure to run a sanity check on existing data first; any row with an unexpected status will block the migration. Effort: 10 minutes including sanity check. M7 Cascade behavior — soft-delete works, hard-delete still breaks Location: backend/src/models/orgQueries.js line 67. deleteOrganization now propagates soft-deletes across users , agent_identities , api_keys , webhooks , and policy_rules . Good. The user-facing delete flow is correct. What's still wrong: actual DELETE FROM organizations WHERE id = ... (e.g. from a maintenance script or an admin tool) will fail with FK constraint errors because no FK has ON DELETE CASCADE . The migration didn't add cascading FKs. This isn't a vulnerability, it's an operational landmine. The first time someone tries to hard-delete an org from psql, they'll get a confusing error. Fix during a maintenance window: ALTER TABLE users DROP CONSTRAINT users_org_id_fkey, ADD CONSTRAINT users_org_id_fkey FOREIGN KEY (org_id) REFERENCES organizations(id) ON DELETE CASCADE; -- Repeat for agent_identities, api_keys, webhooks, policy_rules, audit_logs, agent_groups Or document that hard deletion is unsupported and route admins to soft-delete only. Effort: 30 minutes including migration safety review. M10 Webhook events default — service still allows empty events array Location: backend/src/services/webhookService.js lines 180-184. The route validation now requires events to be a non-empty array. But the service-side filter still defaults to &quot;match all events&quot; if the events array is empty in the DB: const matchingWebhooks = result.rows.filter((webhook) =&gt; { if (!webhook.events || (Array.isArray(webhook.events) &amp;&amp; webhook.events.length === 0)) { return true; // matches every event type } // ... }); For data going through the API, this is moot now. Effort: 15 minutes. M13 Failed login audit — only logs when user exists Location: backend/src/routes/auth.js lines 188-203. The failed-login audit logging is conditional on if (user) : if (user) { try { // increment counter and lock } catch (...) { ... } try { await logAction({ orgId: user.org_id, ... }); } catch (...) { ... } } Good for known users. But credential stuffing typically targets email lists where most addresses don't exist in your system. Those attempts hit the bcrypt check (correctly, for timing reasons) but produce zero audit signal. From the SOC's perspective, an attacker probing 10,000 emails sees nothing in the audit log unless 1+ of them happen to be real. Fix: log a separate login_attempt_unknown_user action when the user doesn't exist, with no actor_id and no org_id . This needs a different audit destination (since the org-scoped audit table requires org_id ), or a system-level audit table. Either way, the visibility gap is the issue. if (!user) { await logUnknownUserLoginAttempt({ email: email.toLowerCase().trim(), ip: req.ip, userAgent: req.get('user-agent') }); } Effort: 1-2 hours including the system-level audit table or a dedicated auth_attempts table. L1 Docker compose secrets — JWT fixed, others still default to devpassword Location: docker-compose.yml . JWT_SECRET: ${JWT_SECRET:?JWT_SECRET must be set} is correct. Compose now fails without the env var. POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-devpassword} and REDIS_PASSWORD: ${REDIS_PASSWORD:-devpassword} still default to a known value. For local dev that's intentional, but if anyone copies this file to a production setup and forgets to set the env vars, the DB password is devpassword and your Postgres is reachable on the same Docker network from a compromised container. Two ways to fix this depending on your goal: For &quot;this is dev only, don't ship it to prod&quot;: rename to docker-compose.dev.yml and add a real docker-compose.yml with no defaults. New issues introduced by the fixes These are bugs or risks in the patch itself, not pre-existing items. N1 CSRF middleware blocks API key callers without X-Requested-With Location: backend/server.js lines 108-120. The CSRF check fires on all mutating requests except the public-path bypass list. There's no carve-out for Bearer-auth requests. So API key callers (CLI tools, CI bots, server-to-server integrations) must include X-Requested-With: AgentID on every request. This is documentation friction more than a bug, but it's the kind of thing that breaks integrations the day after deployment. CSRF doesn't apply to bearer auth (the browser doesn't auto-attach the credential), so the cleaner pattern is: app.use((req, res, next) =&gt; { if (['POST', 'PUT', 'DELETE', 'PATCH'].includes(req.method)) { // Public bypass const publicPaths = ['/public/', '/badge/', '/widget/', '/health']; if (publicPaths.some(p =&gt; req.path.startsWith(p))) return next(); // Bearer auth doesn't need CSRF (browsers don't auto-attach) const authHeader = req.headers.authorization; if (authHeader &amp;&amp; authHeader.startsWith('Bearer ')) return next(); if (req.get('X-Requested-With') !== 'AgentID') { return res.status(403).json({ error: 'CSRF validation failed' }); } } next(); }); Severity: low. Either fix this or update API docs to require the header. Effort: 5 minutes. N2 Locked-account timing leak Location: backend/src/routes/auth.js lines 161-165. The original H2 fix correctly added a dummy bcrypt for missing users. The lockout code introduced a new fast path that bypasses bcrypt entirely: if (user &amp;&amp; user.locked_until &amp;&amp; new Date(user.locked_until) &gt; new Date()) { return res.status(423).json({ error: 'Account temporarily locked...' }); } const hashToCompare = user ? user.password_hash : DUMMY_HASH; const valid = await comparePassword(password, hashToCompare); The fast path returns ~1ms. The slow path takes ~100ms. An attacker timing the response can determine whether an account is currently locked. Combined with the lock threshold (5 failures), this gives attackers free intel: lock the account by trying any 5 passwords, then time future requests to see when the 15-min window resets. Useful for: targeted DoS (keep an account locked indefinitely by re-triggering before unlock), or determining if your target account has been brute-forced recently. Fix: always run bcrypt. Move the lockout check to after the password comparison, or run a dummy bcrypt before the early return: if (user &amp;&amp; user.locked_until &amp;&amp; new Date(user.locked_until) &gt; new Date()) { await comparePassword(password, DUMMY_HASH); // burn the time return res.status(423).json({ error: '...' }); } Severity: low-medium. Information leak, not a credential bypass. Effort: 10 minutes. N3 API key auth doesn't filter deleted users Location: backend/src/middleware/authenticate.js lines 52-61. The new JOIN to fetch the user's role: SELECT ak.id, ak.org_id, ak.user_id, ak.scopes, u.role FROM api_keys ak JOIN users u ON ak.user_id = u.id WHERE ak.key_hash = $1 AND ak.revoked_at IS NULL AND (ak.expires_at IS NULL OR ak.expires_at &gt; NOW()) There's no AND u.deleted_at IS NULL clause. If a user is soft-deleted (whether via the org delete cascade or directly), their API keys keep working until manually revoked. Fix: WHERE ak.key_hash = $1 AND ak.revoked_at IS NULL AND (ak.expires_at IS NULL OR ak.expires_at &gt; NOW()) AND u.deleted_at IS NULL This is also a problem in the org delete cascade: when an org is soft-deleted, users get deleted_at set, but their API keys are revoked separately in the same transaction. Fine for the cascade path, but if a user is deleted individually (e.g. via a future &quot;remove user&quot; admin action), their API keys remain active. Severity: medium. Privilege persistence after user deletion. Effort: 15 minutes. N4 Hash chain depends on JSON.stringify key ordering Location: backend/src/services/auditService.js line 60. buildHashPayload uses JSON.stringify on an object literal with explicit key order. JavaScript guarantees object key insertion order for string keys, so this is deterministic at the top level. But the changes and metadata fields are nested objects whose key order is whatever the producer happened to use: return JSON.stringify({ org_id: ..., // ... changes: entry.changes, // &lt;-- nested object, keys in insertion order metadata: entry.metadata, // &lt;-- same // ... }); If changes is {name: 'foo', desc: 'bar'} on insert and {desc: 'bar', name: 'foo'} on verify (because of a database driver quirk, a JSON column reordering, or a producer-side change), the hash differs and verifyAuditChain returns invalid. This isn't exploitable as a vulnerability — an attacker can't cause this to misreport in their favor — but it's a correctness fragility that will show up the day someone restores from a backup or a future Postgres version changes JSONB key handling. Fix: use a deterministic stringifier. const stringify = require('safe-stable-stringify'); function buildHashPayload(entry) { return stringify({ org_id: entry.orgId || entry.org_id || null, // ... }); } Both logAction and verifyAuditChain should use the same stringifier (they already share buildHashPayload , so this is a one-line change). Severity: low. Future correctness issue. Effort: 30 minutes including a test for ordering invariance. Where things stand The 43-finding patch landed clean. The remaining items are: 1 medium-severity behavioral regression (C6 legacy fallback) that needs a real cutoff 1 medium DNS rebinding gap (C1) that needs IP pinning 1 medium privilege persistence after user delete (N3) A handful of small cleanups (C3 owner-by-pubkey, M3 GET /api-keys, M6 status CHECK, M7 hard-delete cascade, M8 demo flag, M10 service-side default, M13 unknown-user audit, L1 PG/Redis defaults, N1 CSRF + bearer, N2 timing on locked accounts, N4 deterministic stringifier) Effort to clear all remaining items: roughly 8-12 dev hours. None of them are exploit-of-the-week material. C6 is the only one I'd want closed before any high-stakes customer goes live. The fix quality on what was addressed is good. Patterns are consistent, error handling is proper, the new utility (urlValidator) is reusable, and the schema migration is idempotent. The audit hash chain coverage in particular went from &quot;claimed but bypassable&quot; to &quot;actually correct&quot; — that's a meaningful security upgrade. Two pieces of broader feedback: The legacy fallback pattern in C6 is a smell that's worth examining elsewhere. Anywhere you have &quot;old format still works&quot; without a deprecation deadline, you've created the same shape of bug. A grep for console.warn.*[Dd]eprecat will find them. The defense-in-depth gaps (M10 service-side default, N3 deleted user filter, M6 status CHECK) all fall in the same pattern: the route validates correctly but the service or the database doesn't enforce the invariant. For a security product, the database should be the last word on what's possible. CHECK constraints, partial indexes with WHERE clauses, NOT NULL defaults. Make the database refuse the bug, then the application can't accidentally allow it. End of follow-up.
