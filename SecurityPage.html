The Core Guarantee

AgentID never asks for your private key — not in the registration form, not in the API, not anywhere. We verify you own your keypair by asking you to sign a one-time challenge. The signature proves ownership. The private key never moves.

⚡
How Verification Works
The challenge-response flow — same model used by SSH and Solana
1
AgentID Issues a Challenge Random Nonce
We generate a cryptographically random one-time message tied to your pubkey and a 5-minute expiry. This nonce has never been used before and never will be again.
AGENTID-VERIFY:{your_pubkey}:{random_nonce}:{timestamp}
2
Your Agent Signs Locally Private Key Stays Home
Your agent signs the challenge message using its private key — entirely on your own machine or server. The private key never leaves your environment.
signature = Ed25519.sign(challenge_message, YOUR_private_key) → send back only the signature, not the key
3
AgentID Verifies the Signature Public Key Only
We verify the signature using only your public key — a mathematical operation that confirms the signature was produced by the matching private key without ever seeing it.
valid = Ed25519.verify(signature, challenge_message, public_key) nonce consumed → cannot be replayed
4
Registration Complete ✓
If the signature is valid, your agent is registered. The nonce is permanently deleted. The process cannot be replayed — even with the same signature.

Attack Scenarios
What an attacker can try and why it fails
🔴 Register as an agent you don't own
✓ Blocked
An attacker knows InfraWatch's public key and tries to register as InfraWatch. They cannot produce a valid Ed25519 signature without the matching private key — an operation requiring 2¹²⁸ attempts to brute force.
Attacker submits: pubkey = DVKMPX3... (InfraWatch's real pubkey) AgentID issues: "Sign: AGENTID-VERIFY:DVKMPX3:RANDOM_NONCE:TIMESTAMP" Attacker tries: fabricated signature AgentID checks: Ed25519.verify(fakeSig, message, pubkey) Result: ❌ REJECTED — Invalid signature
🔴 Replay a previously captured signature
✓ Blocked
An attacker intercepts a valid challenge-response exchange and tries to replay it to impersonate the agent at a later time.
Attacker captures: valid {pubkey, nonce: "a7f3k9", signature} from previous exchange Attacker replays: same data 10 minutes later AgentID checks: Redis.get("challenge:DVKMPX3:a7f3k9") → NOT FOUND (consumed + expired) Result: ❌ REJECTED — Nonce not found or expired
🔴 Extract private keys from AgentID's database
✓ Not Possible
An attacker gains full read access to the PostgreSQL database and attempts to find private keys to impersonate agents.
Attacker dumps: SELECT * FROM agent_identities; Attacker finds: pubkeys, names, scores, descriptions Attacker looks for: private_key column Result: ❌ COLUMN DOES NOT EXIST — Private keys are never stored anywhere in AgentID
⚠️ Submit false flags to damage an agent's reputation
Known — Sprint 2 Fix
An attacker submits multiple flags against a legitimate agent claiming false reporter pubkeys. The flag submission endpoint currently does not require cryptographic proof that the reporter owns the claimed pubkey. This is a known issue being addressed in Sprint 2.
Attacker POSTs: { agentPubkey: "DVKMPX3...", reporterPubkey: "MADE_UP", reason: "scam" } Current behavior: Flag is accepted (reporter pubkey not verified) Sprint 2 fix: Require Ed25519 signature from reporter proving pubkey ownership After fix: ❌ Rejected without valid reporter signature

Security Best Practices for Agent Operators
How to keep your agent identity secure on your end
🔑 Generating Your Keypair
  • ✓
    Generate on a machine you control — never use an online generator
  • ✓
    Use the provided AgentID keygen script or a trusted tool
  • !
    Never share your private key in support chats, screenshots, or forms
🗄️ Storing Your Private Key
  • ✓
    Store in a .env file on your server — verify it's in .gitignore
  • ✓
    Use a secrets manager (Doppler, AWS Secrets Manager) for production
  • !
    Never commit .env to GitHub — treat it like your master password
🚨 If Your Key is Compromised
  • ✓
    Generate a fresh keypair immediately and register a new agent identity
  • ✓
    Flag your old agent identity noting the compromise
  • !
    Contact the AgentID team to request suspension of the compromised pubkey
🤖 For Autonomous Agents
  • ✓
    One keypair per agent — never share keypairs across services
  • ✓
    Check registration status on startup — re-register with a new key if needed
  • !
    Monitor your reputation score for unexpected drops — may indicate false flag activity
